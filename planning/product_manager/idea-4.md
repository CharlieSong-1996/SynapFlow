# SynapFlow 需求方案 - 处理器节点执行类型与增量语义（V1）

## 1. 背景与目标
在既有“数据节点 + 处理器节点”的模型下，为处理器节点建立统一的执行类型体系，明确：
- 上游数据发生增删改时的调度行为。
- 节点内部是否允许记忆。
- 可提供的单元测试能力与质量保障边界。

目标是让用户在创建 Processor 时即可预判性能特征、正确性风险和重算成本。

## 2. 本次确认的关键决策
1. 处理器节点划分为四类：Strict Mapping、Addition Only、Recreation Only、Self Managed。
2. 所有类型在运行时都以统一事件语义接入（Add / Update / Delete / Reset），但各类型可声明不同的支持矩阵。
3. Addition Only 节点除“输入顺序无关性测试”外，额外开放“幂等性测试”。
4. Self Managed 节点开放“输入无序测试”与“增删等价测试”。
5. 当 Processor 的处理程序发生变更但 Meta Data 未变化时，系统需提供“是否重算历史数据”的用户选项。
6. Self Managed 节点若支持历史版本处理逻辑切换，必须实现 Manage History API。
7. Processor 的 Meta Data 编辑与处理程序编辑必须显式分离；Chat Agent 不直接编辑 Meta Data，Meta Data 编辑 Agent 在专属对话中工作。

## 3. 类型定义（V1）
### 3.1 Strict Mapping
- 核心语义：输入与输出条目严格可追踪；支持一条输入对应零/一/多条输出，但每条输出仅可归属一个输入。
- 状态模型：无内部记忆。
- 执行 API：脚本函数只接受单条输入记录。
- 变更响应：上游增删改可精确定位受影响输入并做局部重算。

### 3.2 Addition Only
- 核心语义：节点维护内部记忆，仅支持“新增写入”与“清空重建”。
- 状态模型：有内部记忆。
- 执行 API：
  - append(record) 或等价新增接口。
  - clear() 或等价清空接口。
- 变更响应：
  - 上游 Add：执行增量处理。
  - 上游 Update/Delete：触发 clear + 全量重放。

### 3.3 Recreation Only
- 核心语义：不维护记忆，任何输入集合变化都走全量重算。
- 状态模型：无内部记忆。
- 执行 API：接受完整输入集合并输出完整结果。
- 变更响应：Add/Update/Delete 统一退化为 Reset 后全量计算。

### 3.4 Self Managed
- 核心语义：节点维护内部记忆，并自行消费上游全部变更事件，实现最优更新策略。
- 状态模型：有内部记忆。
- 执行 API：需支持 Add/Update/Delete/Reset 的事件处理入口。
- 历史管理：当用户允许历史数据沿用新版本逻辑或要求保留旧版本结果时，节点需通过 Manage History API 自主管理历史状态迁移、回放或隔离。
- 变更响应：由节点内部算法决定重算粒度与输出更新策略。

## 4. 统一事件协议与调度策略
### 4.1 事件定义
- Add：上游新增记录。
- Update：上游记录更新。
- Delete：上游记录删除。
- Reset：要求节点清空并以当前输入快照重建输出。

### 4.2 调度映射（框架侧）
| 类型 | Add | Update | Delete | Reset |
|---|---|---|---|---|
| Strict Mapping | 增量处理 | 增量处理 | 增量处理 | 支持（通常用于恢复） |
| Addition Only | 增量处理 | 转换为 Reset | 转换为 Reset | 支持 |
| Recreation Only | 转换为 Reset | 转换为 Reset | 转换为 Reset | 支持 |
| Self Managed | 透传 | 透传 | 透传 | 透传 |

## 5. 测试能力矩阵（V1）
### 5.1 基础测试（所有类型）
- 冒烟执行：脚本无异常。
- 超时保护：在限定时间内完成。
- 输出可序列化与 Meta 一致性检查。

### 5.2 类型专项测试
| 测试项 | Strict Mapping | Addition Only | Recreation Only | Self Managed |
|---|---|---|---|---|
| 输入顺序无关性 | 可选 | 可选 | 可选 | 可选 |
| 幂等性测试 | 可选 | 可选（已确认开放） | 可选 | 可选 |
| 输入无序测试（事件乱序/分批） | - | - | - | 可选（已确认开放） |
| 增删等价测试 | - | - | - | 可选（已确认开放） |

说明：
- 幂等性测试：相同输入事件重复投递后，输出不产生额外副作用。
- 输入无序测试：同一批逻辑事件以不同到达顺序输入，结果满足定义一致性。
- 增删等价测试：在满足约束前提下，某些 Add+Delete 组合与“无操作”对最终状态等价。

## 6. 处理程序版本与历史数据策略（V1）
### 6.1 触发条件
- 当 Processor 的处理程序发生变更，但输出 Meta Data 未变化时，系统不应默认假设必须重算全部历史数据。
- 此时由用户明确选择历史版本处理策略。

### 6.2 用户可选策略
1. 仅新数据使用新逻辑：已有历史输出保持不变，后续新进入的数据使用新版本处理程序。
2. 全量历史重算：框架对当前可见历史输入重新执行处理程序，并刷新输出。
3. 保留版本分界：系统记录某个版本切点，切点前后数据分别由不同版本逻辑产生，供用户追溯。

### 6.3 对不同类型的影响
| 类型 | 支持历史不重算 | 支持全量重算 | 支持版本分界 |
|---|---|---|---|
| Strict Mapping | 支持 | 支持 | 支持 |
| Addition Only | 有条件支持 | 支持 | 有条件支持 |
| Recreation Only | 不建议 | 支持 | 不建议 |
| Self Managed | 支持 | 支持 | 支持（需 Manage History API） |

说明：
- Addition Only 是否支持“仅新数据使用新逻辑”取决于其内部记忆是否允许安全跨版本延续；默认应保守提示风险。
- Recreation Only 的核心心智是“输入集变了就整体重算”，因此不建议强调历史分界能力。

### 6.4 Self Managed 的 Manage History API
Self Managed 需额外暴露历史管理接口，用于处理逻辑版本切换带来的状态迁移问题。最小职责包括：
- 标记当前状态所属的逻辑版本。
- 决定历史状态是迁移、冻结、丢弃还是回放重建。
- 在版本分界模式下，保证下游可追溯输出来自哪个逻辑版本。

## 7. 编辑职责与 Agent 边界（V1）
### 7.1 Meta Data 编辑
- Meta Data 包括输入输出结构、列定义、类型说明、可视化标记等声明性信息。
- Meta Data 编辑必须在专属的 Meta Data 编辑流程或专属 Agent 对话中完成。
- Chat Agent 不直接修改 Meta Data。

### 7.2 处理程序编辑
- 处理程序编辑由 Chat Agent 辅助完成，聚焦脚本逻辑、算法与测试。
- 当处理程序修改不影响 Meta Data 时，可直接进入“历史数据是否重算”的选择流程。
- 当处理程序修改导致 Meta Data 变更时，必须切换到 Meta Data 编辑流程，不能由 Chat Agent 直接越权处理。

## 8. 画布交互与用户旅程（MVP）
1. 用户创建 Processor，首先选择执行类型。
2. 用户通过专属流程定义或修改 Processor Meta Data。
3. 系统根据类型自动生成脚本模板与 API 骨架，由 Chat Agent 辅助用户编辑处理程序。
4. 用户连接上游后，配置面板展示该类型的“变更处理策略说明”。
5. 当用户修改处理程序且 Meta Data 未变化时，系统弹出“历史数据策略选择”：仅影响新数据、全量历史重算、保留版本分界。
6. 用户进入 Test 模式，系统展示基础测试 + 类型专项测试开关。
7. 用户发布后，在运行日志中可查看每次变更走了增量路径还是 Reset 路径、使用了哪个程序版本以及触发原因。

## 9. 设计与可用性建议
1. 类型选择时附带“性能-复杂度”提示：
   - Strict Mapping：简单、可预测。
   - Addition Only：新增快、删改贵。
   - Recreation Only：最稳定、成本最高。
   - Self Managed：性能潜力最高、实现门槛最高。
2. 对触发 Reset 的情况提供可解释提示（例如“收到 Delete，类型不支持删除增量处理”）。
3. 测试开关默认最小化展开，避免一次暴露过多高级概念。
4. 对 Self Managed 提供事件可视化回放，辅助定位状态机错误。
5. 处理程序编辑界面与 Meta Data 编辑界面在视觉上应有明确边界，避免用户误以为二者可被同一 Agent 同时修改。
6. 当用户选择“不重算历史”时，UI 应明确提示历史输出与新输出可能来自不同处理逻辑版本。

## 10. 验收标准（MVP）
1. 创建 Processor 时必须选择一种执行类型。
2. 运行时调度严格符合本方案的类型映射矩阵。
3. Addition Only 可在 Test 模式开启幂等性测试。
4. Self Managed 可在 Test 模式开启输入无序测试与增删等价测试。
5. 当处理程序变更且 Meta Data 不变时，系统必须要求用户明确选择历史数据策略。
6. Self Managed 在启用历史版本管理时，必须具备可调用的 Manage History API。
7. Chat Agent 无法直接编辑 Meta Data；Meta Data 仅能通过专属编辑流程完成。
8. 运行日志可解释每次重算为何走增量或全量路径，并标注处理程序版本。

## 11. 待确认边界（下一轮）
1. Strict Mapping 的多输出条目是否要求稳定顺序。
2. Addition Only 触发 Reset 后，是否默认级联触发下游全链路重算。
3. Self Managed 是否要求强制声明幂等保证级别（例如至少一次投递容忍）。
4. “保留版本分界”是否需要在输出数据节点层面直接展示版本标签。
